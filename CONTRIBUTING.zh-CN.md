# 贡献指南

感谢你对 ArduFlux 项目的关注！提交修改前请阅读以下规范。

## 文档规范（强制要求）

**必须严格遵守双语言 README 同步要求：**
- 本项目维护两份 README 文件：`README.md`（英文）和 `README.zh-CN.md`（中文）。
- 对任意一份 README 的内容修改，**必须**同步应用到另一份文件，保持结构一致、技术信息准确。
- 除非明确标注为语言专属内容，否则不要仅在单个语言版本中添加信息。
- 新增面向所有用户的文档文件时，请遵循相同的命名约定：英文版 `<name>.md`，中文版 `<name>.zh-CN.md`。

## 开发流程

1. Fork 本仓库，从 `master` 分支创建你的特性分支。
2. 搭建开发环境：
   ```bash
   npm install
   npm run watch
   ```
3. 在 VS Code 中按 `F5` 启动扩展开发宿主窗口进行测试。

## 代码规范

- 已启用 TypeScript 严格模式（`tsconfig.json` 中 `strict: true`），无明确理由禁止使用 `any` 类型。
- 保持函数短小，职责单一。
- 使用语义明确的变量和函数名，避免含义模糊的缩写。
- 注释仅用于解释**为什么**要做出非显而易见的决策，而不是解释代码在做什么。

## 测试要求

- 所有新功能或 Bug 修复**必须**配套对应的单元测试。
- 提交前运行完整测试套件：
  ```bash
  npm test
  ```
- 测试文件位于 `src/test/` 目录，按模块分组。

## 提交信息规范

遵循 [约定式提交](https://www.conventionalcommits.org/) 格式，使用中文书写：

- `feat:` 新功能
- `fix:` 修复 Bug
- `refactor:` 代码重构（既不修 Bug 也不加新功能）
- `test:` 增加或更新测试
- `chore:` 构建工具或辅助工具的变动
- `docs:` 文档更新

示例：
```
feat(串口): 新增波特率自动检测功能
```

## Pull Request 规范

- PR 范围应聚焦，单个 PR 只包含一个特性或修复。
- 清晰描述改动内容以及改动的背景和动机。
- 请求 review 前请确保所有 CI 检查已通过。
